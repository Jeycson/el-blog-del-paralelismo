<h2 id="introduccion">Introducción</h2>

El análisis de rendimiento en sistemas de cómputo paralelos representa un pilar fundamental para optimizar algoritmos de procesamiento digital de señales e imágenes. Históricamente, el incremento de rendimiento se ligaba al escalado de frecuencia del reloj en microprocesadores mononúcleo (Ley de Moore). Sin embargo, las barreras físicas impuestas por la disipación térmica y el consumo energético (Wall Térmico) reorientaron la evolución del hardware hacia arquitecturas multinúcleo y unidades de procesamiento masivamente paralelas. 

En este contexto, la paralelización a nivel de hilos de software (TLP, *Thread-Level Parallelism*) y la paralelización a nivel de instrucciones (ILP, *Instruction-Level Parallelism*) mediante unidades vectoriales surgen como las estrategias dominantes. El presente reporte evalúa empíricamente el comportamiento del algoritmo bajo un escenario de carga de trabajo constante (Tamaño de Imagen Fijo) variando la densidad de hilos concurrentes. El objetivo es identificar experimentalmente las fronteras de escalabilidad lineal, cuantificar el impacto del fenómeno de *overhead* por sincronización en el planificador del sistema operativo y contrastar la eficiencia de entornos nativos compilados frente a lenguajes interpretados.

<h2 id="metodologia">Metodología</h2>

El diseño experimental se fundamenta en un análisis cuantitativo y comparativo bajo un enfoque de escalamiento fuerte (Strong Scaling), donde el tamaño total del problema se mantiene invariante (`W = constante`) mientras se incrementan de forma controlada los recursos de procesamiento (`p = 2, 4, 8, 16` hilos). 

Para mitigar la varianza estocástica introducida por factores externos del entorno de ejecución (procesos en segundo plano del sistema operativo, interrupciones de hardware y políticas de ahorro energético), se implementó un protocolo de muestreo estadístico basado en 100 réplicas independientes por cada configuración experimental. Los estimadores analizados para caracterizar la distribución de probabilidad del tiempo de ejecución (`T`) incluyen el límite inferior o tiempo óptimo (T_min), el tiempo promedio como métrica de tendencia central (T_promedio), y el límite superior o peor escenario registrado (T_max), este último crítico para evaluar la previsibilidad del sistema en entornos de tiempo real.

<h3 id="configuracion">Configuración</h3>

Las pruebas se desplegaron sobre una arquitectura de hardware x86_64 dotada de núcleos físicos hiper-entrelazados (Hyper-Threading). El entorno de software y compilación se configuró bajo las siguientes especificaciones:

**Implementación OpenMP (C):** Código fuente nativo con soporte para multiprocesamiento de memoria compartida (TLP). Distribuye los fragmentos del bucle de forma equitativa mediante una directiva estática y sincroniza los hilos a través de una barrera implícita al finalizar la región paralela.
```
void procesar_paralelo_omp(BMP& bmp) {
    BYTE* data = bmp.getImageData();
    int size = bmp.getImageSize();

#pragma omp parallel for schedule(static) num_threads(NUM)
    for (int i = 0; i < size; i++) {
        int temp = data[i] + 50;
        if (temp > 255) data[i] = 255;
        else if (temp < 0) data[i] = 0;
        else data[i] = (BYTE)temp;
    }
}
```

**Implementación SIMD (C):** Enfoque híbrido que combina paralelismo a nivel de hilos (TLP) y de datos (DLP). Segmenta el buffer de la imagen en fragmentos proporcionales al número de hilos definidos (NUM) mediante la API nativa de Windows (CreateThread). Cada hilo invoca una rutina secundaria encargada de ejecutar instrucciones intrínsecas vectoriales para procesar múltiples píxeles por ciclo de reloj utilizando un arreglo de control (u).
```
void procesar_paralelo_simd(BMP& bmp, unsigned char u[16]) {
    HANDLE hThread[NUM] = { NULL };
    LPDWORD deTheradId = { NULL };
    STDATOS datos[NUM];

    for (int i = 0; i < NUM; i++)
    {
        datos[i].gimg = bmp.getImageData() + (sizeof(BYTE) * (bmp.getImageSize() / NUM) * i);
        datos[i].size = bmp.getImageSize() / NUM;
        memcpy(datos[i].u, u, 16);
        hThread[i] = CreateThread(NULL, 0, (LPTHREAD_START_ROUTINE)hilo, &datos[i], 0, deTheradId);

        if (hThread[i] == NULL) {
            printf("Error al crear el hilo %d\n", i);
            return;
        }
    }

    for (int i = 0; i < NUM; i++)
    {
        WaitForSingleObject(hThread[i], INFINITE);
        CloseHandle(hThread[i]);
    }
}
```

<h2 id="conclusiones">Conclusiones</h2>

Con base a las hipotesis, podemos afirmar que las respuestas de Jairo y Jeo son las que mejor predicen el comportamiento real del sistema, siendo la de Jairo la más exacta: “Los datos demuestran que añadir hilos no mejora el rendimiento indefinidamente; por el contrario, existe un punto de inflexión óptimo (4 hilos para OpenMP y Python, y 2 hilos para SIMD)“, donde se obtienen los menores tiempos promedio (4.89 ms, 8.56 ms y 4.48 ms respectivamente). A partir de ahí, al subir a 8 y 16 hilos, el rendimiento empeora y los tiempos se elevan. Esto ocurre porque el tamaño de la imagen es fijo y el trabajo se fragmenta tanto que el sistema operativo gasta más tiempo y recursos en la administración, sincronización y "cambios de contexto" de los hilos (overhead), que en el procesamiento real de los píxeles de brillo.

En el apartado de software, Windows debe gestionar la sobrecarga de hilos, lo que dispara la inestabilidad. Esto se evidencia de forma crítica en los "peores tiempos" de la tabla: con 8 hilos, OpenMP sufre un pico de retraso de 12.48 ms y Python se dispara hasta unos alarmantes 39.89 ms, bloqueado además por su GIL (Global Interpreter Lock), que impide el verdadero paralelismo multifilamento en la CPU.

Para romper esta barrera física y de software, la estrategia más efectiva no es añadir más hilos, sino incrementar el tamaño del problema (Escalamiento de Gustafson); si se procesa una imagen mucho más grande, el tiempo de cómputo por hilo justificará plenamente el costo de organizarlos. Como dato curioso y contundente que arrojan las métricas, la tecnología SIMD en C con solo 2 hilos logra un tiempo excelente de 4.48 ms, superando incluso a OpenMP con 2 hilos (5.25 ms); esto demuestra que el aprovechamiento de los vectores de la propia CPU es inicialmente más limpio y eficiente que saturar prematuramente el sistema con paralelismo a nivel de hilos.
