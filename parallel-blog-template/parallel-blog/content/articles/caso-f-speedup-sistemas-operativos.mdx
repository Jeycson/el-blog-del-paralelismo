<h2 id="introduccion">Introducción</h2>

El análisis de rendimiento en sistemas de cómputo paralelos es fundamental para optimizar algoritmos de procesamiento digital de imágenes. Ante las limitaciones físicas de los procesadores modernos, la evolución del hardware se ha volcado en el desarrollo de arquitecturas multinúcleo. Esto obliga a diseñar software capaz de aprovechar el paralelismo a nivel de hilos (TLP) y el paralelismo a nivel de datos (DLP) mediante unidades vectoriales.

Este reporte evalúa de forma práctica el comportamiento de un algoritmo con una carga de trabajo constante (tamaño de imagen fijo) al incrementar progresivamente la cantidad de hilos concurrentes (2, 4, 8 y 16). El objetivo primordial es resolver **qué va a ocurrir con el *SpeedUp* en cada sistema operativo a medida que se satura el entorno con más hilos**. A través de un análisis comparativo cruzado entre Windows y Linux, se busca identificar el punto de inflexión exacto donde el rendimiento decae debido al *overhead* (sobrecarga) de sincronización y determinar cómo influyen los planificadores de hilos y las restricciones de software en la eficiencia real del sistema.

<h2 id="metodologia">Metodología</h2>

El diseño experimental se fundamenta en un análisis cuantitativo y comparativo bajo un enfoque de **escalamiento fuerte (Strong Scaling)**. En este modelo, la carga de trabajo total se mantiene invariante (tamaño de imagen fijo) mientras se incrementan de forma controlada los recursos de procesamiento físico y lógico en un rango de p = 2, 4, 8, 16 hilos. Este enfoque permite aislar el comportamiento del factor de aceleración (SpeedUp) y evaluar directamente el impacto del incremento de hilos en cada sistema operativo.

Para resolver la hipótesis central y analizar con rigor los límites de escalabilidad, el protocolo experimental recolecta de forma simultánea dos métricas fundamentales por cada tecnología evaluada (Python, OpenMP en C y SIMD en C):
* **Tiempo de ejecución absoluto (T):** Registrado en milisegundos (ms) para determinar la latencia real de procesamiento.
* **Factor de aceleración absoluto (SpeedUp):** Calculado de forma estricta al contrastar el mejor tiempo registrado en la ejecución secuencial o base frente al tiempo obtenido en la configuración paralela con p hilos.

<h3 id="configuracion">Configuración</h3>

Las pruebas se desplegaron sobre una arquitectura de hardware x86_64 dotada de núcleos físicos hiper-entrelazados (Hyper-Threading). El entorno de software y compilación se configuró bajo las siguientes especificaciones:

**Entorno Operativo:** Microsoft Windows 11, operando bajo un esquema de planificación de hilos de prioridad dinámica.

**El codigo base para la ejecucion en C:** Infraestructura principal en C diseñada para la carga de la imagen, inicialización del vector de ajuste de brillo, temporización de los enfoques paralelos y guardado del resultado final.

**Implementación OpenMP (C):** Código fuente nativo con soporte para multiprocesamiento de memoria compartida (TLP). Distribuye los fragmentos del bucle de forma equitativa mediante una directiva estática y sincroniza los hilos a través de una barrera implícita al finalizar la región paralela.
```
void procesar_paralelo_omp(BMP& bmp) {
    BYTE* data = bmp.getImageData();
    int size = bmp.getImageSize();

#pragma omp parallel for schedule(static) num_threads(NUM)
    for (int i = 0; i < size; i++) {
        int temp = data[i] + 50;
        if (temp > 255) data[i] = 255;
        else if (temp < 0) data[i] = 0;
        else data[i] = (BYTE)temp;
    }
}
```

**Implementación SIMD (C):** Enfoque híbrido que combina paralelismo a nivel de hilos (TLP) y de datos (DLP). Segmenta el buffer de la imagen en fragmentos proporcionales al número de hilos definidos (NUM) mediante la API nativa de Windows (CreateThread). Cada hilo invoca una rutina secundaria encargada de ejecutar instrucciones intrínsecas vectoriales para procesar múltiples píxeles por ciclo de reloj utilizando un arreglo de control (u).
```
void procesar_paralelo_simd(BMP& bmp, unsigned char u[16]) {
    HANDLE hThread[NUM] = { NULL };
    LPDWORD deTheradId = { NULL };
    STDATOS datos[NUM];

    for (int i = 0; i < NUM; i++)
    {
        datos[i].gimg = bmp.getImageData() + (sizeof(BYTE) * (bmp.getImageSize() / NUM) * i);
        datos[i].size = bmp.getImageSize() / NUM;
        memcpy(datos[i].u, u, 16);
        hThread[i] = CreateThread(NULL, 0, (LPTHREAD_START_ROUTINE)hilo, &datos[i], 0, deTheradId);

        if (hThread[i] == NULL) {
            printf("Error al crear el hilo %d\n", i);
            return;
        }
    }

    for (int i = 0; i < NUM; i++)
    {
        WaitForSingleObject(hThread[i], INFINITE);
        CloseHandle(hThread[i]);
    }
}
```

El dato analítico más destacado revela que **Python en Linux reportó el SpeedUp relativo más alto de todo el experimento (1.66 con 4 hilos)**, superando porcentualmente a las tecnologías nativas en C.

Sin embargo, este comportamiento representa un **"falso espejismo"** de rendimiento: dado que su código base con 2 hilos en Linux era extremadamente lento e ineficiente (52.64 ms), al asignarle 4 hilos el kernel redujo ese tiempo a 31.64 ms, disparando matemáticamente la aceleración relativa. En tiempos absolutos de ejecución, Python sigue siendo la tecnología más rezagada por una diferencia abismal frente a las versiones en C (OpenMP y SIMD), las cuales mantuvieron registros siempre inferiores a un solo milisegundo en Linux, a pesar de sufrir caídas en su SpeedUp por saturación.

<h2 id="conclusiones">Conclusiones</h2>

Con base en el análisis teórico y los datos empíricos recolectados, se valida que las hipótesis planteadas por Jairo y Mariana predijeron con total precisión el comportamiento real del experimento, seguidas de cerca por las observaciones de Jeovani y Jeycson respecto al estancamiento del sistema. El fenómeno físico quedó demostrado bajo el argumento de Jairo: al mantener un tamaño de imagen fijo y reducido, la porción de carga de trabajo asignada a cada hilo se vuelve insignificante. Esto provoca que el sistema operativo invierta más tiempo y recursos en la creación, administración y alternancia de los hilos (fase de overhead) que en el procesamiento real de los píxeles, validando las curvas descendentes anticipadas por Mariana bajo la premisa de que un exceso de hilos degrada la eficiencia.

El hallazgo analítico más destacable del estudio radica en el comportamiento de Python bajo el entorno Linux, el cual reportó el factor de aceleración relativo más alto de todo el experimento (un SpeedUp de 1.66 utilizando 4 hilos), superando porcentualmente a las soluciones nativas en C (OpenMP y SIMD). Sin embargo, este fenómeno se cataloga estrictamente como un **"falso espejismo"** de rendimiento. Al evaluar los tiempos absolutos, la configuración base de Python con 2 hilos en Linux demostró una profunda ineficiencia latente (52.64 ms); por lo tanto, la reducción del tiempo a 31.64 ms al implementar 4 hilos disparó de forma matemática el cálculo de aceleración respecto a su propio estado inicial. En términos absolutos de ejecución, Python permanece como la tecnología más rezagada por una diferencia abismal frente a las implementaciones en C, las cuales mantuvieron registros siempre inferiores a un solo milisegundo a pesar de sufrir caídas críticas en sus curvas de SpeedUp.

En conclusión, bajo un esquema de escalamiento fuerte con una carga de trabajo pequeña e invariable, el SpeedUp alcanzará un punto de inflexión óptimo muy temprano (fijado en 4 hilos para la mayoría de entornos) y decaerá drásticamente al saturar la arquitectura con 8 y 16 hilos. Los resultados demuestran que el planificador de procesos de Linux gestiona la distribución de hilos de forma más agresiva y eficiente sobre los núcleos físicos en comparación con la política conservadora y la contención del GIL observadas en Windows 11. No obstante, la sobrecarga por hilos termina afectando a ambos sistemas operativos por igual, empujando los coeficientes de OpenMP y SIMD por debajo de la barrera de la eficiencia elemental, lo que demuestra que la fragmentación excesiva de un problema pequeño no justifica el uso de paralelismo multifilamento masivo.
